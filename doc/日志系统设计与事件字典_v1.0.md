## 0. 关键澄清结论（你指出的 10 项）

### 0.1 “不可丢”与“不影响主流程”冲突（阻塞）

决议：

1. 主流程优先，不阻塞游戏回合逻辑（单次日志写入预算 `<= 5ms`）。
2. `AuditLog` 定义为“正常资源下不可丢”，不是“故障态绝对不丢”。
3. 故障降级顺序：
   1. 本地磁盘写入失败 -> 切换内存缓冲（有界 RingBuffer）。
   2. 缓冲溢出 -> 先丢 `DiagLog`，再丢 `DecisionLog`。
   3. 仅在极端故障下允许丢 `AuditLog`，但必须写 `log.audit.drop` 告警计数。

### 0.2 事件 ID 必填/选填边界（阻塞）

决议：按事件阶段定义可空规则，见 4.3《ID 字段必填矩阵》。

### 0.3 `seq` 并发来源未定义（阻塞）

决议：

1. 仅“规范日志管道（Core Logger）”可以分配 `seq`。
2. `WebUI test-hooks` 事件不直接作为规范日志；必须经过映射层后再分配 `seq`。
3. `seq` 使用 `Interlocked.Increment`，作用域为 `round_id`（单局单调递增）。

### 0.4 `reason_code` 与 bool API 断层（阻塞）

决议：

1. 先定义统一结果结构 `OperationResult`（见 7.2）。
2. 日志实现前，规则/流程关键 API 先提供 `...Ex()` 结果版本（保留旧 bool 版本做兼容）。
3. 规范日志只记录可追溯 `reason_code`，禁止落 `UNKNOWN_ERROR` 作为常态。

### 0.5 `payload` 与 `metrics` 分工不一致（阻塞）

决议：

1. `payload` 仅放业务语义字段（状态、对象、结果）。
2. `metrics` 仅放数值/性能字段（如 `elapsed_ms`, `candidate_count`）。
3. `perf.*` 事件也遵循该规则；`elapsed_ms` 必须在 `metrics`。

### 0.6 事件命名与现有实现不一致（阻塞）

决议：

1. 规范事件名统一 `dot.case`，如 `game.start`、`trick.finish`。
2. 现有 `snake_case`（如 `game_start`）保留在 test-hooks，仅作为“原始测试事件”。
3. 通过映射层统一转换，见 8.4《命名映射表》。

### 0.7 阶段模型与规则文档不一致

决议：采用“规范超集阶段”，并提供当前核心阶段映射，见 6.1。

### 0.8 `game_id` 与 `round_id` 语义

决议：

1. `game_id`：整场升级赛（Campaign）ID。
2. `round_id`：单局（25 墩 + 结算）ID。
3. 当前实现若尚无 Campaign，可暂设 “1 game_id 对应 1 round_id”，但字段保留双层结构。

### 0.9 保留策略时间口径

决议：

1. 存储与清理口径统一 UTC。
2. 展示层按本地时区换算。
3. 清理任务失败需补偿重试，见 9.3。

### 0.10 卡牌字段编码统一

决议：

1. 规范日志卡牌字段使用对象，不使用 `"♠K"` 裸字符串。
2. 必须包含 `card_instance_id` 以区分双副牌同花色同点数。
3. 标准卡牌结构见 5.2。

## 1. 目标与范围

本设计用于单机 Web/Unity 共用核心（Core）日志方案，目标如下：

1. 支持完整对局复盘（谁在何时做了什么，为什么失败）。
2. 支持规则、UI、AI 问题快速定位。
3. 支持 AI 策略效果分析与性能分析。
4. 保证低侵入、可扩展、可降级。

适用范围：

1. `Core`：发牌、亮主、扣底、出牌、计分、升级、AI 决策。
2. `WebUI`：关键交互点击、提示、异常。
3. 暂不包含：远端日志服务接入。

---

## 2. 设计原则

1. 统一事件模型：所有规范日志统一 JSON 信封格式。
2. 业务可复盘优先：`AuditLog` 正常资源下不采样、不丢失。
3. 主流程优先：日志系统异常不得阻塞主流程。
4. 调试可控：`DiagLog` 可按环境降采样。
5. 性能可观测：关键路径必须记录耗时指标。

---

## 3. 日志分层

### 3.1 审计日志（AuditLog）

用途：业务复盘与争议追踪。  
约束：正常资源下不采样；故障态允许受控丢失并告警。

### 3.2 决策日志（DecisionLog）

用途：AI 候选集、评分、选择路径分析。  
约束：生产可采样，默认全量。

### 3.3 诊断日志（DiagLog）

用途：开发调试、异常上下文。  
约束：可采样、可动态调级。

### 3.4 性能日志（PerfLog）

用途：慢调用定位（AI 决策、规则校验、整墩结算）。  
约束：指标字段统一放 `metrics`。

## 4. 统一事件信封（Envelope Schema）

### 4.1 标准结构

```json
{
  "schema_version": "1.1",
  "ts_utc": "2026-03-13T10:35:11.239Z",
  "level": "INFO",
  "category": "audit|decision|diag|perf",
  "event": "play.attempt",
  "session_id": "sess_20260313_9f2a",
  "game_id": "game_000128",
  "round_id": "round_0007",
  "trick_id": "trick_0019",
  "turn_id": "turn_0075",
  "seq": 1276,
  "phase": "PlayTricks",
  "actor": "player_0|ai_2|system|ui",
  "correlation_id": "corr_6cf4",
  "causation_id": "evt_1275",
  "payload": {},
  "metrics": {}
}
```

### 4.2 字段总规则

1. `schema_version` 固定 `1.1`。
2. `event` 必须来自 8《事件字典 v1.0》。
3. `seq` 仅由规范日志管道分配（非前端直写）。
4. `payload` 只放业务语义字段。
5. `metrics` 只放数值与性能字段。

### 4.3 ID 字段必填矩阵

| 事件范围 | session_id | game_id | round_id | trick_id | turn_id | seq |
|---|---|---|---|---|---|---|
| `session.*` | 必填 | 选填 | 选填 | 选填 | 选填 | 必填 |
| `game.*` | 必填 | 必填 | 选填 | 选填 | 选填 | 必填 |
| `round.*` / `phase.*` | 必填 | 必填 | 必填 | 选填 | 选填 | 必填 |
| `dealing.*` / `trump.*` / `bury.*` | 必填 | 必填 | 必填 | 选填 | 选填 | 必填 |
| `turn.*` / `play.*` / `trick.*` | 必填 | 必填 | 必填 | 必填 | 必填 | 必填 |
| `score.*` / `level.*` | 必填 | 必填 | 必填 | 选填 | 选填 | 必填 |
| `ui.*`（规范化后） | 必填 | 选填 | 选填 | 选填 | 选填 | 必填 |
| `error.*` | 必填 | 选填 | 选填 | 选填 | 选填 | 必填 |

## 5. 数据编码规范

### 5.1 命名规范

1. 规范日志事件名：`dot.case`。
2. 字段名：`snake_case`。

### 5.2 卡牌标准结构（统一）

```json
{
  "card_instance_id": "S-K-1",
  "suit": "Spade",
  "rank": "King",
  "score": 10,
  "text": "♠K"
}
```

字段说明：

1. `card_instance_id`：必须存在，用于区分双副牌中的重复牌。
2. `text`：仅用于可读展示，解析以 `suit/rank/card_instance_id` 为准。

### 7.2 结果结构（实现前置）

建议统一结构：

```csharp
public record OperationResult(
    bool Success,
    string? ReasonCode = null,
    Dictionary<string, object>? Detail = null
);
```

实现策略：

1. 关键流程 API 先增加 `...Ex()` 返回 `OperationResult`。
2. 旧 bool API 保留，内部调用 `...Ex()`，兼容现有调用方。

## 9. 落盘与保留策略

### 9.1 文件组织

```
logs/
  raw/
    2026-03-13/
      tractor-2026-03-13-00.jsonl
  archive/
    2026-03-13/
      tractor-2026-03-13.jsonl.gz
```

### 9.2 保留策略（固定）

1. 原始日志（`raw`）保留 3 天。
2. 压缩日志（`archive`）保留 14 天。
3. 第 3 天将原始日志压缩归档后删除原始文件。
4. 第 14 天删除压缩归档。

### 9.3 时间口径与补偿

1. 存储与清理统一使用 UTC。
2. 展示给用户时再转换本地时区（如 `Asia/Hong_Kong`）。
3. 清理任务每日 `00:10 UTC` 执行。
4. 若任务失败，下一次执行需补偿扫描（幂等删除）。
