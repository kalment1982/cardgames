# WebUI Review页面简版设计 v1.0

## 1. 文档目的

本文件用于冻结 `http://127.0.0.1:5168/review` 页的第一阶段实现口径。

当前阶段目标不是一次性做完整分析平台，而是先把 `Review` 页调到：

- 页面主布局稳定
- 与玩家正常对局页视觉结构基本一致
- 可以逐墩复盘
- 可以查看 4 家手牌
- 可以通过弹窗查看额外 AI 决策信息

第一阶段先保持：

- 仅默认加载 `1` 局样本
- 先把页面结构、交互和视觉调试通过
- 页面稳定后，再放开完整会话列表加载

---

## 2. 本阶段总体原则

### 2.1 与游戏页保持一致

`Review` 页主舞台应尽量复用当前玩家对局页的空间布局与视觉感受，包括：

- 中央出牌区位置关系一致
- 东西南北玩家方位一致
- 手牌显示风格尽量一致
- 牌面图片、大小、整理逻辑尽量一致
- 底色和桌面氛围尽量一致

目的：

- 让用户在 `Game` 页和 `Review` 页之间切换时没有认知成本
- 用户可以直接把某一墩和真实对局画面对应起来

### 2.2 主界面只放“复盘时必须一直看到”的信息

主界面优先展示：

- 本墩牌桌状态
- 4 家手牌
- 当前墩出牌结果
- 当前墩赢家、得分、首发
- 当前座位 AI 线路标记（`V21 / V30`）

不应把大量 bundle / trace / rule tag 直接堆在主界面，否则会破坏“像真实牌桌”的可读性。

### 2.3 额外信息统一进弹窗

AI 决策的详细解释信息放到弹窗，不直接常驻在牌桌区域。

触发方式：

- 主界面保留一个统一按钮，例如：`额外信息`
- 点击后弹出当前墩详情
- 如有需要，也允许点击单个玩家区域，打开该玩家的详情弹窗

---

## 3. 主界面结构

## 3.1 顶部工具区

保留但尽量简化，只放操作性内容：

- 上一墩
- 下一墩
- 跳墩
- 查看/隐藏底牌
- 额外信息
- 复制问题定位

建议保留的信息：

- 当前 round / trick uid
- 当前墩编号
- 闲家分变化
- 当前墩赢家
- 本墩得分

建议不在顶部长条区直接展示的内容：

- 完整 decision trace
- phase policy
- triggered rules
- 大段 warning 文本

这些应进入弹窗或折叠区。

## 3.2 中央牌桌区

应尽量对齐当前 `GamePage` 的牌桌布局：

- 北家在上
- 西家在左
- 东家在右
- 南家在下
- 中间是本墩出牌区

### 每个座位主界面上固定展示

- 方位名：`东 / 西 / 南 / 北`
- AI 标识：`V21` 或 `V30`
- 庄家标记：如果该家是庄家则显示 `庄`
- 当前手牌数量

### 推荐展示形式

例如：

- `北 · V30`
- `东 · V21`
- `南 · V30 · 庄`

---

## 4. 手牌展示要求

## 4.1 四家都显示手牌

这是 `Review` 页与正常对局页最关键的差异之一。

主界面需要直接看到 4 家手牌：

- 南家：横向展开
- 北家：横向展开
- 西家：竖向双列
- 东家：竖向双列

## 4.2 手牌要整理

整理规则应尽量与当前玩家对局页一致，至少满足：

- 主牌优先
- 王优先
- 同花色聚合
- 同花色内部按牌力排序
- 级牌在主牌规则下正确归类

这点非常重要，因为 `Review` 页的核心任务之一是判断：

- 某家当时有没有更好的牌可出
- 某家是否浪费了大牌
- 某家是否应该继续走某门副牌

如果手牌不整理，复盘效率会明显下降。

## 4.3 东西家横向多行

东西两家如果按竖向单列或竖向双列展示，容易把南家区域挤掉，Safari 下也更容易出现溢出。

因此当前冻结为：

- 东家手牌按横向多行显示
- 西家手牌按横向多行显示
- 每行内部允许重叠
- 一行放不下再换到下一行

这样更符合 `一屏内同时看全四家手牌` 的目标。

## 4.4 当前冻结的牌面显示参数

为避免后续多人调 UI 时出现漂移，`Review` 页牌面显示先冻结这一组参数：

- 主页面不再使用左侧 round 常驻列表，round 通过顶部下拉切换
- 北/南手牌：横向重叠显示
- 东/西手牌：横向多行显示，单行内重叠，放不下再换行
- 首发标记：当前墩 `PlayOrder = 1` 用高亮色并显示 `首`

当前统一牌面尺寸基线：

- 手牌：`48 x 70`
- 当前墩出牌：`46 x 68`
- 底牌：`50 x 74`

当前统一重叠基线：

- 北/南手牌重叠：`-24px`
- 东/西手牌单行重叠：后续以北/南重叠视觉为参考继续收敛
- 当前墩出牌区：后续如需叠牌展示，也以北/南手牌重叠视觉为基准

后续若继续调整：

- 优先改一套统一参数，不单独手调某一个座位
- 主目标始终是：`一屏内看全四家手牌 + 当前墩`

## 5. 主界面与弹窗的分工

## 5.1 主界面保留的信息

主界面保留：

- 当前墩 4 家手牌
- 当前墩实际出牌
- 当前墩赢家
- 当前墩得分
- 当前墩首发
- 各家 AI 线路标识
- 底牌查看按钮

这些信息能支撑绝大多数“这一手有没有更好出法”的判断。

## 5.2 弹窗中展示的信息

弹窗中展示：

- phase
- path
- phase policy
- primary intent
- secondary intent
- selected reason
- selected candidate id
- turn id
- decision trace id
- triggered rules
- AI 选牌
- 实际出牌
- 是否一致
- v30 额外字段

如果后续还要继续扩展，优先加到弹窗，不要先加到主舞台。

## 5.3 日志告警的位置

`warnings` 建议不常驻主界面中部。

更适合的做法：

- 主界面只显示一个告警计数
- 详细 warning 列表进弹窗

原因：

- warning 文本通常偏长
- 常驻会挤压牌桌空间
- 会破坏“先看牌，再看解释”的复盘顺序

---

## 6. 第一阶段确认保留的限制

第一阶段先保留：

- 只加载 `1` 局样本
- 先调通单局完整复盘
- 暂不做大列表性能优化
- 暂不做批量筛选和复杂搜索

页面调通后，再逐步放开：

- 多局列表
- 完整搜索
- 来源过滤
- 规则/意图筛选

---

## 7. 当前方案没有明显问题，但有两点需要注意

## 7.1 牌桌区不要被信息面板挤爆

如果把：

- meta 信息
- warning
- decision 详情
- 规则标签

都放到主界面，会导致牌桌区域过小，最终最重要的“出牌判断”反而不好看。

因此本版必须坚持：

- 主界面看牌局
- 弹窗看解释

## 7.2 手动复盘时最核心的是“实际牌面”

用户真正要判断的是：

- 当时 4 家各自有什么牌
- 已经出了什么牌
- 这一墩谁赢了
- 某个 AI 有没有更好的牌

所以第一优先级不是把所有 JSON 字段铺出来，而是把：

- 手牌布局
- 牌型整理
- 实际出牌区

做对、做稳、做清楚。

---

## 8. 第一阶段实施建议

按下面顺序落地：

1. 保持只加载 1 局
2. 把 `Review` 页主舞台改成和 `GamePage` 一致的牌桌布局
3. 把 4 家手牌全部接入主界面
4. 完成南北横排、东西双列竖排
5. 给四家加 `V21 / V30 / 庄` 标识
6. 把详细 AI 信息收进弹窗
7. 页面调试稳定后，再放开完整局列表

---

## 9. 本版结论

本阶段 `Review` 页的正确方向是：

- 不是做“数据后台”
- 而是做“可逐墩复盘的真实牌桌”

主界面看牌面和出牌结果，弹窗看 AI 解释，这个分工是合理的，可以直接作为第一阶段实现口径。
