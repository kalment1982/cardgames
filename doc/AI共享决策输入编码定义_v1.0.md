# AI共享决策输入编码定义 v1.0

## 1. 文档定位

本文档是 `AI共享决策输入定义_v1.0.md` 的配套编码文档。

本文档只回答 4 个问题：

1. 每个共享输入项使用什么编码形式
2. 哪些输入需要保留顺序信息
3. 哪些输入需要归一化
4. PPO observation 如何按阶段拼接

本文档不重复定义字段语义。字段含义、阶段边界与 RuleAI -> PPO 映射关系，以：

- [AI共享决策输入定义_v1.0](./AI共享决策输入定义_v1.0.md)

为准。

---

## 2. 编码设计原则

统一遵守以下原则：

1. 同一输入项在不同 AI 中语义一致，编码允许不同，但不能改变含义
2. 阶段 1 以低噪声、易实现、强相关为先，不追求一次性最优编码
3. 阶段 2 在阶段 1 基础上增量扩展，不重做 observation 主体结构
4. 原始牌面信息优先使用牌面计数编码，避免过早引入复杂手工摘要
5. 顺序会影响决策的输入，必须显式保留顺序信息
6. 枚举类和角色类字段优先使用 one-hot 或小范围离散编码
7. 数量类字段优先使用 count / scalar，并在 PPO 侧做有限归一化

---

## 3. 编码形式说明

| 编码形式 | 说明 | 适用场景 |
|---|---|---|
| `CardCount108` | 108 维牌面计数编码；每一张实体牌按固定索引累计计数 | 自己手牌、已出牌、当前牌桌牌面 |
| `CardSet54x2` | 以 54 种牌面 * 2 副牌的固定索引表示 | 与 `CardCount108` 语义一致，可作为实现细节 |
| `OneHotN` | N 维 one-hot 编码 | 角色、主花色、级牌、阶段标签 |
| `Scalar` | 单个数值标量 | 墩分、主牌数量、对子数、剩余手牌数 |
| `NormalizedScalar` | 归一化后的单值标量 | 进度、比分、剩余牌数等范围明确字段 |
| `PerPlayerVector` | 按 4 家展开的定长向量 | 各玩家缺门、各玩家主牌估计、各玩家剩余手牌 |
| `PerSuitVector` | 按 4 门副牌展开的定长向量 | 各花色长度、缺门信息、被毙风险 |
| `OrderedSlots` | 保留出牌先后顺序的固定槽位结构 | 当前墩已出牌、按顺序展开的牌桌状态 |
| `Mask` | 与动作空间等长的合法动作掩码 | PPO action mask |

## 4. 阶段1编码总表

| 共享输入项 | 推荐编码 | 是否保序 | 是否归一化 | 编码说明 | 备注 |
|---|---|---|---|---|---|
| 自己手牌 | `CardCount108` | 否 | 否 | 按固定牌索引统计自己当前手牌 | 阶段1核心原始输入 |
| 自己角色 | `OneHot3` | 否 | 否 | 庄家 / 庄家队友 / 闲家 | 若实现上沿用 `AIRole`，需映射到稳定三分类 |
| 自己座位 | `OneHot4` | 否 | 否 | 4 家座位 one-hot | 与庄家座位结合使用 |
| 庄家座位 | `OneHot4` | 否 | 否 | 4 家座位 one-hot | 不建议只存相对座位，先保留绝对座位 |
| 主花色/无主 | `OneHot5` | 否 | 否 | 黑桃/红桃/梅花/方块/无主 | 必须显式编码无主 |
| 当前级牌 | `OneHot13` | 否 | 否 | A-K 或项目内 rank 顺序稳定映射 | 需与正式规则口径一致 |
| 当前第几墩 | `NormalizedScalar` | 否 | 是 | 以总墩数上限做归一化 | 反映局面阶段 |
| 当前出牌位置 | `OneHot4` | 否 | 否 | 首手/二手/三手/四手 | 顺序影响大 |
| 当前是否首发 | `Scalar` | 否 | 否 | 0/1 布尔值；可由 `play_position` 推导，但建议显式提供 | 减少 Python 侧推断逻辑 |
| 当前墩首发牌 | `CardCount108` | 是 | 否 | 只编码首发那一手的牌面 | 牌型信息可从牌面恢复 |
| 当前领先牌组 | `CardCount108` | 是 | 否 | 编码当前暂时最大的一组牌 | 跟牌/毙牌判断核心 |
| 当前赢家 | `OneHot5` | 否 | 否 | 4 家 + “暂无赢家” | 推荐保留“无”状态 |
| 队友是否暂时领先 | `Scalar` | 否 | 否 | 0/1 布尔值 | 高价值语义压缩 |
| 当前墩累计分数 | `NormalizedScalar` | 否 | 是 | 建议按单墩理论上限归一化 | 高分墩判断基础 |
| 当前闲家得分 | `NormalizedScalar` | 否 | 是 | 即 `defender_score`，建议按总分上限归一化 | 当前比分态势输入 |
| 各玩家剩余手牌数 | `PerPlayerVector` | 否 | 是 | 4 维向量，建议按单人手牌上限归一化 | 进度与末局感知 |
| 主牌数量 | `NormalizedScalar` | 否 | 是 | 按最大可能主牌数归一化 | 结构统计 |
| 高主数量 | `NormalizedScalar` | 否 | 是 | 按合理上限归一化 | 结构统计 |
| 王数量 | `Scalar` | 否 | 否 | 0/1/2/3/4 | 范围小，可不归一 |
| 级牌数量 | `Scalar` | 否 | 否 | 当前手里级牌张数 | 范围小，可不归一 |
| 主牌对子数量 | `Scalar` | 否 | 否 | 主牌对子组数 | 范围小 |
| 主牌拖拉机数量 | `Scalar` | 否 | 否 | 主牌拖拉机组数 | 范围小 |
| 各花色长度 | `PerSuitVector` | 否 | 可选 | 4 维副花色长度向量 | 建议归一化到单门上限 |
| 手牌分牌数量 | `Scalar` | 否 | 否 | 当前手里分牌张数 | 与墩分一起看 |

## 5. 阶段1不引入项的编码策略

阶段 1 明确不引入的输入项，在编码层统一处理为：

1. 不进入 PPO observation
2. 不在中间向量中预留假字段
3. 不用 0 填充伪装成“已接入”

原因：

- 防止“看起来字段已经有了，实际没有语义”
- 防止阶段 1 和阶段 2 observation 结构混乱

## 6. 阶段2新增编码总表

| 共享输入项 | 推荐编码 | 是否保序 | 是否归一化 | 编码说明 | 备注 |
|---|---|---|---|---|---|
| 已出牌计数 | `CardCount108` | 否 | 可选 | 按固定牌索引统计全局已出牌张数 | 阶段2记牌基础 |
| 各玩家缺门信息 | `PerPlayerVector + PerSuitVector` | 否 | 否 | 可编码为 4x4 布尔矩阵 | 4玩家 × 4副门 |
| 已知底牌信息 | `CardCount108` | 否 | 否 | 对已知底牌做牌面计数编码 | 庄家与非庄家口径不同需显式区分 |
| 无对子证据 | `PerPlayerVector + PerSuitVector` | 否 | 否 | 可编码为 4x4 布尔证据矩阵 | 证据不是确定事实 |
| 无拖拉机证据 | `PerPlayerVector + PerSuitVector` | 否 | 否 | 可编码为 4x4 布尔证据矩阵 | 结构推断证据 |
| 其他玩家主牌数量估计 | `PerPlayerVector` | 否 | 是 | 每家一个估计值，可先只编码 estimate | 第一版可不带置信区间 |
| 其他玩家高主风险 | `PerPlayerVector` | 否 | 否 | 每家一个离散风险等级 | 可映射成 0/1/2/3 |
| 某体系首发被毙风险 | `PerSuitVector` | 否 | 否 | 按花色/体系展开风险等级 | 第一版可先做 4 维副门风险 |
| 保底压力 | `Scalar` | 否 | 否 | 风险等级离散值 | 阶段2新增高层语义 |
| 掉庄压力 | `Scalar` | 否 | 否 | 风险等级离散值 | 同上 |
| 抠底争夺压力 | `Scalar` | 否 | 否 | 风险等级离散值 | 同上 |
| 分数压力 | `Scalar` | 否 | 否 | 分数压力等级离散值 | 同上 |
| 末局阶段 | `OneHotN` | 否 | 否 | `None / Late / LastTrickRace` 等稳定枚举 | 需与 `EndgameLevel` 一致 |
| 底牌分数 | `NormalizedScalar` | 否 | 是 | 底牌分值归一化 | 配合保底/抠底使用 |

## 7. 顺序保留规则

以下输入必须保留顺序语义，不能只做无序聚合：

| 输入项 | 为什么必须保序 | 推荐做法 |
|---|---|---|
| 当前墩首发牌 | 首发牌定义跟牌约束和牌型体系 | 单独使用 `CardCount108` 编码，不并入当前桌面总牌 |
| 当前领先牌组 | 当前真实比较目标不是“桌面所有牌”，而是当前最大牌组 | 单独使用 `CardCount108` 编码 |
| 当前出牌位置 | 一手、二手、三手、四手的决策完全不同 | `OneHot4` |

以下输入可以先不显式保序：

| 输入项 | 原因 |
|---|---|
| 自己手牌 | 手牌是集合，不需要按摸牌顺序 |
| 已出牌计数 | 阶段2先只关心“哪些牌已见过”，不先追时间顺序 |

## 8. 归一化规则

建议只对范围明确且可能数值波动较大的字段做归一化。

| 输入项 | 建议 |
|---|---|
| 当前第几墩 | 归一化 |
| 当前墩累计分数 | 归一化 |
| 主牌数量 | 归一化 |
| 高主数量 | 归一化 |
| 各花色长度 | 建议归一化 |
| 底牌分数 | 阶段2归一化 |

以下字段不建议归一化：

| 输入项 | 原因 |
|---|---|
| 王数量 | 范围极小，直接整数更直观 |
| 级牌数量 | 范围极小 |
| 主牌对子数量 | 范围极小 |
| 主牌拖拉机数量 | 范围极小 |
| 各类风险等级 | 属于离散语义，不是连续量 |

## 9. PPO Observation 拼接规则

### 9.1 阶段1 observation

推荐按以下顺序拼接：

1. 自己手牌
2. 角色与座位
3. 主花色 / 无主 / 当前级牌
4. 当前第几墩 / 当前出牌位置 / 当前是否首发
5. 当前墩首发牌
6. 当前领先牌组
7. 当前赢家 / 队友是否领先 / 当前墩累计分数 / 当前闲家得分
8. 各玩家剩余手牌数
9. 手牌结构统计

原则：

- 先原始牌面
- 再规则与角色
- 再当前墩上下文
- 最后结构摘要

### 9.2 阶段2 observation

在阶段1 observation 之后追加：

1. 已出牌计数
2. 缺门与结构证据矩阵
3. 已知底牌信息
4. 主牌估计与风险
5. 首发被毙风险
6. 保底 / 掉庄 / 抠底 / 分数 / 末局压力
7. 底牌分数

原则：

- 阶段2只做追加，不打乱阶段1主体顺序

## 10. 与 RuleAI 日志结构的关系

RuleAI 的日志和 decision bundle 可以保留更丰富的结构化对象，不要求完全压平成 PPO observation。

关系如下：

1. `共享输入定义文档` 负责语义一致
2. `共享输入编码文档` 负责 PPO / 中间向量编码方式
3. `RuleAI 决策日志` 可以继续保留结构化对象，以方便排查和回放

换言之：

- RuleAI 日志是“可解释结构”
- PPO observation 是“训练输入向量”
- 二者语义一致，但编码形态可以不同
