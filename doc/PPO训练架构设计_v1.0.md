# 拖拉机 PPO AI 训练文档 v1.0

## 1. 文档定位

本文档是 PPO AI 的第一版实施文档，不是长期完整架构文档。

本版只回答两个问题：

1. PPO 在当前项目里能不能学会打拖拉机出牌
2. PPO 能不能在现有训练环境下逐步逼近 RuleAI

本版不回答以下问题：

1. PPO 是否已经可以替代正式线上 RuleAI
2. 阶段 2 之后是否还要继续扩展为更复杂的训练系统
3. 是否必须改成候选动作评分式策略网络
4. 是否要做 DAgger、复杂联赛和完整晋级系统

结论：

- 本文档是 `MVP 训练方案`
- 目标是 `先验证可行性，再决定是否工程化`

RuleAI 与 PPO 共享的决策输入字段定义、阶段边界与映射口径，以：

- [AI共享决策输入定义_v1.0](./AI共享决策输入定义_v1.0.md)

为准。本文档不重复维护共享输入语义。

---

## 2. 当前结论

基于现有讨论，本版采用以下判断：

1. 第一版训练需要分阶段，但只分 `2 个阶段`
2. 第一版只训练 `PlayTricks`，不训练叫牌、反主、埋底
3. 第一版使用 `Python 训练器 + C# 正式引擎 + stdio JSON bridge`
4. 第一版继续使用 `固定动作索引 + legal mask`
5. 第一版先跑 `纯 PPO`
6. 第一版 `不使用 warm start`
7. RuleAI 在第一版中只作为 `对手`

---

## 3. 前置定义

### 3.1 训练范围

第一版 PPO 只接管：

- `PlayTricks`

以下阶段继续沿用现有规则逻辑或 RuleAI：

- 发牌阶段
- 亮主 / 反主
- 埋底

原因：

- 先把最核心、最复杂、最容易衡量的出牌阶段跑通

### 3.2 对战配置

第一版统一使用以下对战配置：

- `2 个 PPO 一队 vs 2 个 RuleAI 一队`

说明：

1. PPO 控制同一队的两家
2. 对面两家使用 RuleAI
3. 所有胜率指标均以该配置为准

原因：

1. 拖拉机是双人配合游戏
2. 若只控 1 家，队友噪声过大
3. 2v2 更适合验证“整队策略是否有效”

### 3.3 弱基线定义

阶段 1 使用的弱基线定义为：

- `随机合法出牌 AI`

规则：

1. 只从合法动作集合中随机选择
2. 不做策略评分
3. 不做记牌、推断、保底、抠底判断

用途：

- 用来验证 PPO 是否已经学到基本策略，而不是只会随机合法出牌

### 3.4 warm start 默认策略

阶段 1 默认策略：

- `先纯 PPO`
- `不做行为克隆`
- `不做 RuleAI 预训练`

只有在以下条件满足时，才考虑把 warm start 作为第二实验分支：

1. 合法率已经达标
2. 对 RuleAI 胜率长期很低
3. 训练曲线显示 PPO 没有学到有效策略

## 4. 动作空间设计

阶段 1 的动作空间采用共享固定语义 slot 方案：

- `ACTION_DIM = 384`
- 单张 `54`
- 对子 `54`
- 拖拉机 `260`
- 复杂动作保留槽 `16`

动作空间的唯一真值源以：

- [PPO AI_共享动作空间定义_v1.0](./PPO AI_共享动作空间定义_v1.0.md)

为准。

本文档只保留训练层约束，不重复维护 slot 映射细节：

1. 固定动作索引 + legal mask
2. 首发 / 跟牌共用同一套动作空间
3. `slot -> concrete action` 在给定 state 下必须唯一解码
4. 复杂动作超过保留槽容量时，必须显式返回 overflow，不能静默裁剪

---

## 5. 第一版总目标

第一版 PPO 训练线的总目标是：

1. 建立稳定可重复的训练闭环
2. 证明 PPO 能学会拖拉机出牌
3. 证明 PPO 对 RuleAI 存在逼近趋势

第一版不要求：

1. 超过 RuleAI
2. 直接上线替代 RuleAI
3. 直接接入当前 WebUI 对局运行时

## 6. 阶段划分

本版只分两个阶段。

### 6.1 阶段 1：可训练性验证

阶段 1 要回答的问题：

- `PPO 能不能在当前训练环境里学会打拖拉机出牌`

### 6.2 阶段 2：逼近 RuleAI

阶段 2 要回答的问题：

- `PPO 是否已经具备接近 RuleAI 的对抗能力，值得继续投入`

---

## 7. 阶段 1：可训练性验证

### 7.1 阶段目标

阶段 1 的目标不是大幅变强，而是先证明：

1. 训练可以稳定跑起来
2. PPO 能学会合法出牌
3. PPO 不再只是随机打牌
4. PPO 对 RuleAI 有初步逼近趋势

### 7.2 阶段 1 范围

范围固定如下：

1. 只训练 `PlayTricks`
2. 使用 `Python 训练器 + C# 正式引擎 bridge`
3. 使用 `固定动作索引 + legal mask`
4. 使用最小状态集合
5. 使用最小奖励
6. 不使用 warm start
7. 对手固定为 RuleAI

### 7.3 阶段 1 输入字段

阶段 1 的 PPO 输入字段、文字语义、阶段边界与 RuleAI 映射关系，统一以：

- [AI共享决策输入定义_v1.0](./AI共享决策输入定义_v1.0.md)

中的“阶段1必需输入表”“阶段1明确不引入表”“RuleAI -> PPO 映射表”为准。

### 7.4 阶段 1 奖励最终版

阶段 1 采用终局奖励为主的设计，不在一开始引入过多 shaping。

最终奖励公式：

```text
R_final = win_loss_reward + level_reward + score_reward
```

其中：

1. 终局胜负奖励
   - 赢局：`+10`
   - 输局：`-10`
2. 升级奖励
   - `2 * n`
   - `n` 表示本局我方净升级数
3. 得分奖励
   - `0.02 * final_team_score`
   - `final_team_score` 表示本方最终得分

设计原则：

1. 胜负是主目标
2. 升级数是次目标
3. 得分是辅助目标

约束：

1. 阶段 1 不额外加入保底、抠底、高分墩、烧大牌惩罚等 shaping
2. 阶段 1 优先验证“纯 PPO + 终局奖励”能否学起来
3. 得分奖励使用本方最终得分，不使用大量过程分数累计
4. C# 环境只返回 `terminal_result` 原始结算信息，奖励由 Python 训练侧计算

### 7.5 阶段 1 硬指标

阶段 1 的量化目标如下：

1. 训练可稳定完成预设局数，不崩溃
2. 非法动作率 `<= 5%`
3. 对随机合法出牌 AI 胜率 `>= 70%`
4. 对 RuleAI 胜率达到 `10% - 20%`
5. 平均奖励或胜率曲线存在上升趋势

### 7.6 阶段 1 观察性指标

阶段 1 不把“坏牌率下降”作为硬指标，只作为观察项。

执行方式：

1. 每轮评估抽查固定 `10 局` 回放
2. 人工观察是否存在明显低级错误

重点观察：

- 无意义烧大牌
- 明显该垫小却垫大
- 明显能保分却放分
- 明显乱出主

### 7.7 阶段 1 成功标准

满足以下 4 条即可进入阶段 2：

1. 非法动作率 `<= 5%`
2. 对随机合法出牌 AI 胜率 `>= 70%`
3. 对 RuleAI 胜率 `>= 10%`
4. 训练曲线呈上升趋势

### 7.8 阶段 1 失败标准

满足任一条，阶段 1 视为失败：

1. 训练 `2000 局` 后，对 RuleAI 胜率仍 `< 5%`
2. 非法动作率持续 `> 20%`
3. 训练曲线完全不收敛
4. 对随机合法出牌 AI 仍无明显优势

### 7.9 阶段 1 失败后的处理顺序

若阶段 1 失败，按以下顺序排查：

1. 检查环境逻辑、合法动作集合、legal mask 是否有 bug
2. 检查状态编码是否缺失关键字段
3. 检查奖励是否过稀疏或方向错误
4. 在上述问题排除后，再引入轻量 warm start

---

## 8. 阶段 2：逼近 RuleAI

### 8.1 阶段目标

阶段 2 的目标是判断：

- `PPO 是否已经接近 RuleAI，值得继续投入`

这一阶段开始关注：

1. 实战胜率
2. 局面意识
3. 关键场景质量

### 8.2 阶段 2 范围

阶段 2 仍保持整体轻量，不进入重工程化。

允许增加的内容：

1. 补充状态特征
2. 补充少量高价值奖励 shaping
3. 少量自我对弈
4. 必要时尝试轻量 warm start

仍然不做：

1. C# 正式引擎桥接
2. 候选动作评分式架构
3. DAgger
4. 复杂联赛系统

### 8.3 阶段 2 新增输入字段

阶段 2 新增哪些记牌、推断、局势风险输入，统一以：

- [AI共享决策输入定义_v1.0](./AI共享决策输入定义_v1.0.md)

中的“阶段2新增输入表”“RuleAI -> PPO 映射表”为准。

### 8.4 阶段 2 奖励增强

阶段 2 在阶段 1 终局奖励的基础上，可以增加少量高价值 shaping：

1. 保底成功 / 失败
2. 抠底成功 / 失败
3. 高分墩得失

建议高分墩定义为：

- `当前墩分数 > 20`

约束：

1. shaping 只能辅助训练，不能盖过终局胜负目标
2. 阶段 2 的主目标仍然是终局胜负与对 RuleAI 胜率

### 8.5 阶段 2 硬指标

阶段 2 的量化目标如下：

1. 非法动作率 `<= 1%`
2. 对随机合法出牌 AI 胜率稳定 `>= 85%`
3. 对 RuleAI 胜率达到 `30% - 40%`
4. 多个随机 seed 下结果方向一致
5. 训练曲线稳定，不出现长期退化

### 8.6 阶段 2 专项指标

阶段 2 开始建立简单统计脚本，至少统计以下 3 项：

1. 保底成功率
2. 抠底成功率
3. 高分墩保护率

要求：

- 能从训练或对战日志中稳定提取
- 先求稳定可看，不追求复杂分析

### 8.7 阶段 2 观察性指标

继续抽查固定评估回放，重点看是否减少：

- 放跑高分墩
- 无谓烧大牌
- 错误出主
- 明显协同失误

### 8.8 阶段 2 成功标准

满足以下 4 条，即说明 PPO 线值得继续推进：

1. 非法动作率 `<= 1%`
2. 对 RuleAI 胜率 `>= 30%`
3. 专项统计有正向结果
4. 多 seed 下结果稳定

### 8.9 阶段 2 失败标准

满足任一条，则暂不进入更重的工程化阶段：

1. 长期训练后对 RuleAI 胜率仍 `< 20%`
2. 结果对 seed 高度敏感，难以复现
3. 专项统计无明显改进
4. 训练经常策略崩坏

### 8.10 阶段 2 失败后的处理顺序

若阶段 2 失败，按以下顺序处理：

1. 先补状态与奖励
2. 再尝试轻量 warm start
3. 再调整自我对弈比例
4. 若仍无明显改善，再评估是否升级环境或动作表示

## 9. warm start 策略

### 9.1 第一版默认不使用 warm start

默认原因：

1. 先跑纯 PPO，诊断价值最高
2. 可以明确判断 PPO 是否本身可学
3. 避免一开始就把 RuleAI 偏好写进初始权重

### 9.2 什么时候引入 warm start

仅在以下情况下考虑引入：

1. 合法率已达标
2. 对 RuleAI 胜率长期 `< 5%`
3. 训练曲线不显示有效策略提升

### 9.3 warm start 的限制

若后续引入 warm start，只做轻量预热，不做重度模仿。

建议方式：

1. 用 RuleAI 自对弈生成 `1000 - 5000` 局数据
2. 监督学习训练 `5 - 10` 个 epoch
3. 不训练到收敛
4. 预热结束后立即切回 PPO

目的：

- 让网络“大致知道怎么打”
- 不是让网络“完全模仿 RuleAI”

### 9.4 warm start 的实验要求

若引入 warm start，必须保留对照实验：

1. `纯 PPO`
2. `轻量 warm start + PPO`

要求：

- 使用同样训练预算
- 使用同样评估脚本
- 使用同样对战配置

目的：

- 明确 warm start 是否真的有效

---

## 10. 日志与评估要求

第一版不做复杂联赛系统，但必须保留最小可诊断能力。

至少要有：

1. 每局结果日志
2. 训练曲线
3. 固定 benchmark 评估脚本
4. 固定种子
5. 抽样回放

阶段 2 额外要求：

1. 能统计保底成功率
2. 能统计抠底成功率
3. 能统计高分墩保护率

---

## 11. 当前明确不做的事

本版明确后置以下事项：

1. C# 正式引擎训练桥
2. 候选动作评分式网络
3. DAgger
4. 多层联赛与复杂晋级系统
5. 生产级上线架构

原因：

- 这些都属于 `PPO 已证明可行之后` 的工程化工作

---

## 12. 实施顺序

建议按以下顺序落地：

1. 跑通阶段 1 纯 PPO 训练闭环
2. 完成阶段 1 基线评估
3. 若阶段 1 成功，再进入阶段 2
4. 若阶段 1 失败，先排查环境/状态/奖励
5. 必要时再引入轻量 warm start

---

## 13. 本版结论

PPO 第一版训练方案的核心原则是：

1. 先验证，不先工程化
2. 先纯 PPO，不先 warm start
3. 先证明能学，再讨论超越 RuleAI
4. 阶段 1 用最小必需输入和终局奖励跑通闭环
5. 只在必要时逐步增加复杂度

只要阶段 1 和阶段 2 能顺利达标，就说明 PPO 线值得继续投入。
